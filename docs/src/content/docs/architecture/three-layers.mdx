## Overview

GoMode has three layers. Each can be understood independently.

<Image src={architectureDiagram} alt="GoMode Architecture — three layers: JS Worker, zero-copy data exchange via zerobuf, and WASM (Go + Zig ABI)" />

```
CF Request → Worker (JS)
  → writes request as zerobuf tagged values into WASM memory
  → calls handle_zerobuf(reqPtr)
  → Go reads request at fixed offsets, processes, writes response at fixed offsets
  → JS reads response directly from WASM memory
  → CF Response
```

## Layer 1: Zig ABI (`zig-abi/src/`)

Zig compiles to `wasm32-wasi` and exports functions with C ABI. Both Go and JS can call them directly — no serialization, just pointer math on shared WASM linear memory.

### Memory management

```
JS calls zig_alloc(100) → gets pointer 0x1A00 in wasm.memory
JS writes data at 0x1A00 → Go/Zig reads it at 0x1A00
Zero copy. Same memory, different languages reading it.
```

### SIMD batch operations

Zig provides SIMD-accelerated operations for columnar data processing:

```
zig_simd_sum_f64(ptr, count)       → sum of f64 array
zig_simd_dot_f64(a, b, count)      → dot product
zig_simd_scale_f64(ptr, count, s)  → multiply by scalar
zig_simd_minmax_f64(ptr, count)    → min/max in one pass
```

### Host imports

~10 functions in the `gomode_host` namespace: raw sockets, random bytes, time, console, KV. The JS worker implements these. All HTTP parsing, crypto, etc. stays inside WASM.

## Layer 2: CF Worker (`worker/src/`)

**`worker.ts`** — Entry point. Routes requests to direct WASM execution or a Durable Object. Uses zerobuf to write request fields and read response fields directly in WASM memory — no JSON serialization.

**`go-do.ts`** — Durable Object. Instantiates WASM once, keeps it alive. Each request calls the exported `handle()` function.

### Zero-copy data exchange

Request and response schemas are defined once and reused:

```typescript
import { Arena, defineSchema } from "zerobuf";

const RequestSchema = defineSchema(["method", "path"]);
const ResponseSchema = defineSchema(["status", "contentType", "body"]);

// Per request:
const req = RequestSchema.create(arena, { method: "GET", path: "/json" });
const respPtr = exports.handle_zerobuf(req.__zerobuf_ptr);
const resp = ResponseSchema.toJS(arena, respPtr);
// resp.status, resp.body — read directly from WASM memory
```

## Layer 3: Go Handler (`examples/hello-worker/`)

Go exports a `handle_zerobuf()` function that reads request fields from zerobuf tagged value slots and writes response fields at fixed offsets — no JSON anywhere.

```go
//export handle_zerobuf
func handleZerobuf(reqBase uint32) uint32 {
  method := readZBString(reqAddr + 0*16)  // field 0
  path := readZBString(reqAddr + 1*16)    // field 1

  // Write response: [status, contentType, body]
  writeZBI32(respBase + 0*16, 200)
  writeZBString(respBase + 1*16, "application/json")
  writeZBString(respBase + 2*16, `{"message":"Hello"}`)

  return uint32(respBase)
}
```

Go calls Zig ABI functions via `//go:wasmimport`:

```go
//go:wasmimport gomode zig_alloc
func zigAlloc(len uint32) uint32

//go:wasmimport gomode zig_simd_sum_f64
func zigSimdSumF64(ptr uint32, count uint32) float64
```
