## Setup

- **Machine**: Apple Silicon (M-series)
- **Tool**: `wrk -t2 -c10 -d10s`
- **Workload**: Hello world + JSON response handler
- **Wrangler dev** for GoMode (local miniflare runtime)

## Results

| | Native Go | GoMode (Worker) | GoMode (DO) | Std Go WASM |
|---|---|---|---|---|
| **GET / req/sec** | 80,715 | 3,159 | 1,235 | 614 |
| **GET /json req/sec** | 83,182 | 3,181 | 961 | 537 |
| **Latency (avg)** | 0.6ms | 4.5ms | 9.7ms | 78ms |
| **Binary size** | native | 753KB | 753KB | 3.0MB |

> **Note:** GoMode numbers are bottlenecked by wrangler dev (~3.5K req/sec ceiling). Production CF edge numbers will be significantly higher. We'll update these with production benchmarks soon.

## GoMode vs Standard Go WASM

**5.6x faster** throughput with a **4x smaller** binary.

The standard Go compiler produces a 3MB WASM binary with a heavy runtime (goroutine scheduler, full GC, large stdlib). TinyGo produces a 753KB binary with minimal runtime. GoMode uses zerobuf for zero-copy data exchange — no JSON serialization anywhere in the hot path.

## How zero-copy works

GoMode eliminates all serialization between JS and Go:

1. JS writes request fields as tagged values directly into WASM memory (32 bytes for 2 fields)
2. Go reads at fixed offsets — no parsing
3. Go writes response at fixed offsets — no serialization
4. JS reads at fixed offsets — no parsing

Compare to standard Go WASM which requires `JSON.stringify` → encode → copy → `json.Unmarshal` → process → `json.Marshal` → copy → `JSON.parse` on every request.

## Worker vs Durable Object

Worker mode is **2.6x faster** than DO mode because:

- **No DO queue** — requests don't serialize through a single instance
- **Horizontal scaling** — CF spins up multiple isolates, each with its own cached WASM instance
- **Same warm latency** — both have ~1ms per-request cost once WASM is initialized

Use DO mode when you need state (WebSocket sessions, counters, etc.). Use Worker mode for stateless handlers.

## What's measured

These benchmarks run on **wrangler dev** (local miniflare), not production CF edge. The wrangler dev server itself caps around ~3.5K req/sec, so these numbers reflect the wrangler bottleneck — not GoMode's actual throughput limit. Production numbers will differ:

- Higher throughput (native V8 isolates, multi-core, distributed)
- Lower latency (no miniflare simulation overhead)
- Production benchmarks coming soon

## What's NOT measured yet

- **Production CF edge** — deploy benchmarks (coming soon)
- **Zig SIMD acceleration** — not wired up yet. Will show gains on data transforms and batch processing.
- **Columnar data** — bulk data workloads where the Arrow-like layout shines.
- **Async CF bindings** — KV, R2, D1 via Asyncify (not integrated yet).

## Reproduce

```bash
# Install wrk
brew install wrk

# Build WASM
npm run build:go

# Start dev server
npm run dev

# Benchmark
wrk -t2 -c10 -d10s http://localhost:8787/       # Worker mode
wrk -t2 -c10 -d10s http://localhost:8787/json    # Worker mode (JSON response)
wrk -t2 -c10 -d10s http://localhost:8787/do/     # DO mode
wrk -t2 -c10 -d10s http://localhost:8787/do/json # DO mode (JSON response)
```
